# trivy-docker-api

## 1. System Overview 🛡️
The application consists of three core components that work together: a central Scanner Service, the CI/CD Pipeline in your repository, and a Data Store in the cloud.

Trigger: A developer pushes code, triggering a CI/CD Pipeline.

Request: The pipeline packages the repository and sends it to the Scanner Service via an API call.

Scan & Store: The Scanner Service runs a Trivy scan on the code and uploads the full JSON report to the Data Store (S3 bucket).

Acknowledge: The service sends a 200 OK response to the pipeline, which then continues its other jobs.

## 2. Core Components ⚙️
A. The Scanner Service (Docker + Python)
This is the brain of the operation. It's a lightweight web service built to do one job perfectly.

Technology: Docker, Python (with FastAPI), and Trivy.

Function: It exposes a single API endpoint (/scan-and-store). When it receives a code archive, it runs trivy and uploads the resulting report to the cloud bucket.

Key Files:

Dockerfile: Defines how to build the container image.

main.py: The Python web server logic.

requirements.txt: Lists the Python dependencies (fastapi, boto3, etc.).

B. The CI/CD Pipeline Step (Bitbucket)
This is the trigger that integrates the scanner into your development process.

Technology: bitbucket-pipelines.yml.

Function: This YAML step is a "fire-and-forget" script. It uses standard command-line tools (tar and curl) to package the repository and send it to the Scanner Service. It does not wait for scan results or block the pipeline.

C. The Data Store (AWS S3)
This is your system of record for all vulnerability data.

Technology: AWS S3 (or any cloud object storage).

Function: It stores the raw JSON reports generated by Trivy. Each report is saved with a unique name, creating a complete audit trail. This data can later be used for analytics, building dashboards, or security audits.

## 3. Setup and Deployment Steps 📦
Set Up AWS S3:

Create an S3 bucket (e.g., trivy-scanner-test-harry).

Create a dedicated IAM user with programmatic access.

Attach a policy to this user that grants s3:PutObject permissions for your specific bucket.

Save the user's Access Key ID and Secret Access Key.

Build and Deploy the Scanner Service:

Navigate to the scanner-service directory.

Build the Docker image: docker build -t your-image-name .

Push the image to a container registry (like Docker Hub, AWS ECR, or Google GCR).

Deploy the image to a cloud service like AWS App Runner or Google Cloud Run. During deployment, configure the following environment variables:

SCANNER_API_KEY: Your chosen secret key.

S3_BUCKET_NAME: The name of your S3 bucket.

AWS_ACCESS_KEY_ID: The key for your IAM user.

AWS_SECRET_ACCESS_KEY: The secret for your IAM user.

Configure Your Bitbucket Repository:

Go to Repository settings > Pipelines > Repository variables.

Add SCANNER_URL: The URL of your deployed scanner service.

Add SCANNER_API_KEY: The same secret key you set in the service. Mark this as "Secured".

Add the provided YAML step to your bitbucket-pipelines.yml file.

## 4. How to Test and Use 🧪
Once set up, the system runs automatically. The primary way to test is locally before deployment.

Run the Scanner Locally: Use the docker run command with all the necessary -e flags for your API key and AWS credentials.

Bash

docker run -d --name scanner-app -p 8000:8000 \
  -e SCANNER_API_KEY="..." \
  -e S3_BUCKET_NAME="..." \
  -e AWS_ACCESS_KEY_ID="..." \
  -e AWS_SECRET_ACCESS_KEY="..." \
  your-image-name
Simulate a Pipeline Call:

Navigate to any local code repository in your terminal.

Package it: tar -czf test-app.tar.gz .

Send it: curl -X POST -H "X-API-Key:..." -F "file=@test-app.tar.gz" http://localhost:8000/scan-and-store

Verify: Check your S3 bucket. A new JSON report file should appear, confirming the entire workflow is successful.
